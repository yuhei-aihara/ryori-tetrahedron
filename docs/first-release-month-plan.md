# RYORI 初回リリースと最初の1ヶ月

更新日: 2026-07-12  
参照: `deep-research-report (13).md`

## 1. 初回リリースで約束すること

初回リリースの約束は、レシピの量ではなく、料理を決めるまでの体験に置く。

> **30秒で、いつもの料理を少し動かし、理由のある別の一皿に出会える。**

最初の対象は、料理が好きな人よりも「毎日作っているが、選ぶ料理が固定している人」。この人に、四面体の理論を説明するのではなく、次の一連の体験を渡す。

1. いつもの料理を選ぶ
2. 「香ばしく」「軽く」「温かく」「コクを出す」から方向を選ぶ
3. ひと手間 / 別の一皿 / 冒険の距離を選ぶ
4. Before / After、変わったこと、追加材料、味の変化、理由を見る
5. 作るか、料理の地図で近い料理をもう少し見る

四面体は最初の操作画面ではなく、料理にたどり着いた後に「なぜこの料理なのか」「次にどこへ行けるか」を説明するレイヤーとして使う。

## 2. 初回リリースの範囲

### 入れるもの

- ホームの主役を「いつもの料理を少し動かす」にする
- 起点料理5種を、4方向 × 3段階の編集済み変化として提供する
- 料理の地図への遷移
- 既存の食材検索、四面体探索、記録、味覚マップ、チャレンジ、Pro案内
- 料理の理由、味の変化、追加材料、調理時間の表示
- PWA / GitHub PagesでURLを開けば触れる状態

### 初回リリースではやらないもの

- 500件すべてを同じ品質であるように見せること
- 生成的な料理名を増やすこと
- 決済、広告、EC在庫連携
- ユーザー投稿、ランキング、SNS機能
- 四面体を最初から回転させるチュートリアル

500件のカタログは探索の奥行きとして残すが、ホームの主動線では編集済みの料理変化を優先する。次に増やすべき単位は「料理数」ではなく「起点料理からの変化の道筋」である。

## 3. 周知メッセージ

### 表に出す一文

**レシピを探す前に、いつもの料理を少し動かしてみる。**

### 具体例で伝える

- ポテトサラダを、今日はガレットへ
- 冷ややっこを、今日は香ばしい焼き皿へ
- 卵焼きを、今日はスープの中へ

「四面体」「AI」「3D」を最初の広告文にしない。そこを売ると、機能の説明を読ませることになり、料理に興味がない人には届かない。入口では「いつもの料理が別の方向へ動く」ことだけを見せ、四面体は体験後に発見されるものにする。

## 4. 最初の1ヶ月の進め方

### 1週目: 5〜8人のクローズドテスト

対象は、料理を日常的にする人と、両親のようにアプリの説明に頼らず触る人を混ぜる。説明をせず、次の課題だけを渡す。

> 「今夜の料理をひとつ、いつもと少し違う方向へ動かしてみてください」

観察すること:

- 10秒以内に最初の操作を始めるか
- 「何を動かす画面か」を言葉にできるか
- 方向と距離の違いが伝わるか
- 結果を料理として信じられるか
- 四面体を開いたとき、近さや方向を理解できるか
- 保存 / 作る / 記録のどこで止まるか

この週はアクセス数より、迷った瞬間の言葉を残す。テスト後に「四面体とは何か」を聞くのではなく、「さっき何を選んでいましたか」と聞く。

### 2週目: 20〜30人への小規模公開

クローズドテストで一番反応が良かった起点料理を1つに絞り、短い画面録画またはGIFを添えて共有する。URLだけを送らず、必ず一つの課題を添える。

> 「ポテトサラダを、香ばしくするか、軽くするか、どちらか試して感想をください」

集めるもの:

- どの起点料理が選ばれたか
- どの方向が選ばれたか
- どの距離で止まったか
- 料理の妥当性に関する自由コメント
- 実際に作ったか、作らなかった理由

### 3週目: 中身の改善週

反応の多かった起点料理・方向を優先し、10本程度の変化の道筋を追加する。見た目の追加より、次の3点を直す。

- 変化前後の料理が、調理上つながっているか
- 追加材料が現実的で、家にある代替も想像できるか
- 「なぜ合うか」が味・水分・熱・食感の言葉で説明できるか

四面体の表示は、理解できない人がいても失敗と判断しない。料理に到達した後に開かれ、近い料理へ移動する人がいるかを優先指標にする。

### 4週目: 初回リリースと次の約束

公開告知は機能一覧ではなく、料理の変化を見せる3本で行う。

1. ポテトサラダ → じゃがいものローズマリー焼き
2. 冷ややっこ → 豆腐とトマトの焼き皿
3. 卵焼き → きのこのだし煮

発信先は、まず個人のネットワーク、料理をする友人、Instagram / X / Threadsなどの短尺投稿、食事会やワークショップのQRコード。最初から大きな広告を買わず、どの言葉で開かれるかを確認する。

## 5. 初回リリースのKPI

小規模テストでは統計的な成長率より、行動の連続性を見る。目標は暫定値であり、1週目の観察後に更新する。

| 段階 | 見る指標 | 初期の目安 |
|---|---|---:|
| 入口 | 初回操作まで | 10秒以内 |
| 到達 | 起点料理から結果表示まで | 30秒以内 |
| 理解 | 「何を動かしたか」を説明できる | 70%以上 |
| 興味 | 結果から別の方向を試す | 30%以上 |
| 地図 | 四面体 / 料理の地図を開く | 20%以上 |
| 意思 | 保存する | 20%以上 |
| 実行 | 作った / 記録した | 10%以上 |
| 継続 | 7日以内に再訪 | 15%以上 |

最重要指標は「四面体を開いた人数」ではなく、「料理にたどり着いた後、もう一つの料理へ移動した人数」。これはRYORIが検索結果ではなく料理の世界の探索になっているかを示す。

## 6. コンテンツと運用の原則

- 初回は5起点料理 × 4方向 × 3段階を編集品質の基準にする
- 追加は毎週10道筋までに絞り、実際に作れるか確認する
- 料理名・材料・手順・理由は独自に書き、他社本文や写真をコピーしない
- スポンサー料理を扱う場合は、編集記事と広告を明確に分ける
- 健康効果を断定せず、「軽く感じる」「野菜を足しやすい」程度の料理表現に留める
- アレルギーや健康目的を扱う場合は、後から同意・削除・利用目的の導線を設計する

## 7. 収益化の順番

初月は課金を出さない。まず「この探索は人に伝わるか」を検証する。

1. 初月: 無料公開、編集品質と行動データの確認
2. 2〜3ヶ月目: 食材・調味料・調理器具の小さな特集タイアップ
3. 3ヶ月目以降: 保存数、比較、詳細分析、冷蔵庫条件をPro候補にする
4. データが蓄積した後: 食材・小売・調理器具ブランド向けの料理空間インサイトを検討

Pro案内は、保存数の上限、詳細な味覚分析、冷蔵庫提案など、実際に使いたい制約が生まれたタイミングだけに出す。起動直後の課金画面は出さない。

## 8. 初回リリースの判断基準

以下を満たしたら、初回リリースは成立と判断する。

- 料理をしない人でも、説明なしに最初の方向を選べる
- 料理をする人が、結果を「作れそう」と判断できる
- 起点料理と結果料理の間に、調理上の理由がある
- 四面体を開かなくても価値が成立する
- 四面体を開いた人には、近い料理へ進む理由がある
- 反応をもとに、次週のコンテンツ優先順位を決められる

初回リリースの成功は、料理データを大量に持つことではない。ユーザーが「次はこの方向へ行ってみよう」と自分の好みを言葉にできることにある。
